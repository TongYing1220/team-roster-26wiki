# 管理员审核清单

审核 Pull Request 时使用此清单

## 📋 新成员首次提交审核清单

### 1. 目录结构检查
- [ ] 在 `members/` 目录下创建了个人目录
- [ ] 目录命名符合规范（小写字母、数字、连字符）
- [ ] 目录名称有意义（姓名拼音或学号）
- [ ] 包含 `work-log.md` 文件

### 2. 文件内容检查
- [ ] 使用了正确的模板格式
- [ ] 个人信息完整
  - [ ] 姓名
  - [ ] 学号/工号
  - [ ] GitHub 用户名
  - [ ] 邮箱
  - [ ] 加入日期
  - [ ] 角色/职责
- [ ] 至少有一条工作记录
- [ ] 工作记录格式正确
  - [ ] 日期格式：YYYY-MM-DD
  - [ ] 包含工作类型
  - [ ] 工作内容描述清晰
  - [ ] 工作时长已填写

### 3. Git 提交检查
- [ ] Commit 信息符合规范
  - [ ] 格式：`Add: 添加 [姓名] 的工作记录`
  - [ ] 描述清晰
- [ ] 只修改了自己的目录
- [ ] 没有修改其他成员的文件
- [ ] 没有修改项目配置文件（除非有特殊说明）
- [ ] 没有提交不必要的文件
  - [ ] 无临时文件
  - [ ] 无编辑器配置文件
  - [ ] 无系统文件（.DS_Store 等）

### 4. PR 信息检查
- [ ] PR 标题符合规范：`[新成员] 姓名 - 工作记录提交`
- [ ] PR 描述清晰
- [ ] 填写了 PR 模板

### 5. 代码质量检查
- [ ] Markdown 格式正确
- [ ] 没有明显的拼写错误
- [ ] 表格格式正确
- [ ] 链接有效（如果有）
- [ ] 图片可访问（如果有）

## 📝 更新工作记录审核清单

### 1. 内容更新检查
- [ ] 只修改了自己的 `work-log.md`
- [ ] 新增内容格式正确
- [ ] 日期按时间顺序排列
- [ ] 工作内容描述清晰

### 2. Git 提交检查
- [ ] Commit 信息符合规范：`Update: 更新 [日期] 工作记录`
- [ ] 已同步主仓库（检查是否有冲突）
- [ ] 只包含必要的修改

### 3. PR 信息检查
- [ ] PR 标题符合规范：`[更新] 姓名 - YYYY-MM-DD 工作记录`
- [ ] PR 描述说明了更新内容

## 🚫 常见问题及处理

### 问题1：修改了其他成员的文件
**处理方式**：
1. 在 PR 中评论说明问题
2. 要求成员撤销对其他文件的修改
3. 提供正确的操作指导

**评论模板**：
```markdown
感谢你的提交！但我注意到你修改了其他成员的文件。请按以下步骤修正：

1. 撤销对其他文件的修改：
   `git checkout origin/main -- members/其他成员目录/`
2. 重新提交：
   `git add members/你的目录/`
   `git commit -m "Fix: 移除对其他文件的修改"`
   `git push origin main`

如有问题，请随时提问。
```

### 问题2：格式不符合规范
**处理方式**：
1. 具体指出格式问题
2. 提供正确的格式示例
3. 要求修改后重新提交

**评论模板**：
```markdown
你的工作记录内容很好，但格式需要调整：

**问题**：
- [ ] 日期格式应为 YYYY-MM-DD
- [ ] 缺少工作时长字段
- [ ] 表格格式不正确

**正确格式**：
见模板文件 `templates/member-template.md`

请修改后更新 PR，谢谢！
```

### 问题3：Commit 信息不规范
**处理方式**：
1. 说明 Commit 信息规范
2. 可以直接合并（如果内容没问题）
3. 建议下次注意

**评论模板**：
```markdown
内容审核通过！✅

小建议：Commit 信息建议使用规范格式：
- `Add: 添加 [姓名] 的工作记录`
- `Update: 更新 [日期] 工作记录`

这次我会直接合并，下次注意即可 😊
```

### 问题4：包含敏感信息
**处理方式**：
1. 立即在 PR 中评论提醒
2. 要求删除敏感信息
3. 确认删除后再合并

**评论模板**：
```markdown
⚠️ **重要提醒**：你的提交中包含敏感信息（邮箱/电话/密码等）。

请立即：
1. 删除敏感信息
2. 更新提交
3. 确认安全后我会合并

如需帮助，请联系我。
```

### 问题5：文件过大或格式错误
**处理方式**：
1. 说明文件大小限制
2. 建议压缩或使用外部链接
3. 提供解决方案

## ✅ 审核通过标准

满足以下所有条件可以合并：
1. ✅ 目录结构正确
2. ✅ 只修改了自己的文件
3. ✅ 格式基本符合规范
4. ✅ 内容真实、完整
5. ✅ 无敏感信息
6. ✅ 无明显错误

## 🎯 合并后操作

1. **点击 "Merge pull request"**
2. **选择合并方式**：
   - 推荐：Squash and merge（压缩合并，保持历史清晰）
   - 或：Merge pull request（保留所有 commit）
3. **填写合并信息**：
   ```
   Merge: [姓名] 工作记录提交/更新
   
   - 审核通过
   - 内容完整
   ```
4. **确认合并**
5. **在 PR 中留言感谢**：
   ```markdown
   已合并！感谢你的提交 🎉
   
   欢迎随时更新你的工作记录。如有问题，随时在 Issues 中提问。
   ```

## 🔧 管理员常用命令

```bash
# 查看所有成员目录
ls members/

# 统计成员数量
ls members/ | wc -l

# 查看最近的提交
git log --oneline -10

# 查看某个成员的提交历史
git log --all --oneline -- members/成员名/

# 查看文件修改统计
git diff --stat upstream/main

# 本地测试某个 PR
git fetch origin pull/PR编号/head:test-branch
git checkout test-branch
# 测试完成后
git checkout main
git branch -D test-branch
```
